# AI 黑盒回归测试

本目录包含 Cody 项目的 **AI 驱动黑盒回归测试用例**。

这些测试用例以 Markdown 编写，由 **Claude Code**（或其他 AI Agent）执行。目的是在功能迭代后，让 AI 代替人工完成回归测试。

## 设计理念

- **黑盒测试**：不关注内部实现，只通过 CLI / SDK / Web API 等外部接口验证功能
- **AI 可执行**：每个测试用例都包含明确的操作步骤和验证方法，AI 可以直接执行
- **容错验证**：由于 LLM 输出不确定，验证标准是宽松的（关键词包含、文件存在等），而非精确匹配
- **独立隔离**：每个测试用例使用临时目录，不影响项目源码

## 如何使用

### 给 Claude Code 的执行指令

将以下 prompt 发送给 Claude Code，它会自动执行测试：

```
请阅读 ai_tests/ 目录下的测试用例，按照以下流程执行回归测试：

1. 先阅读 ai_tests/setup.md，向我询问 LLM 配置（CODY_MODEL、CODY_MODEL_API_KEY、CODY_MODEL_BASE_URL），然后完成环境检查
2. 按目录顺序执行每个 .md 文件中的测试用例
3. 每个测试用例：
   a. 阅读「前置条件」，准备环境（通常是创建临时目录）
   b. 按「操作步骤」逐步执行
   c. 按「验证方法」检查结果
   d. 记录 PASS / FAIL 及原因
4. 所有用例执行完毕后：
   a. 在对话中输出测试报告
   b. 将报告写入 ai_tests/reports/<日期>_report.md（如 2026-03-09_report.md）
   c. 如果同一天多次执行，用 _report_2.md, _report_3.md 后缀区分

注意：
- 所有文件操作在 /tmp/cody_ai_test_<timestamp>/ 下进行，测试结束后清理
- 每次执行前必须向用户要 LLM 配置，不要假设环境变量已设置
- 遇到 FAIL 时继续执行后续用例，不要中断
- 报告文件保存在 ai_tests/reports/ 目录下，方便版本追溯
```

### 测试报告格式

AI 执行完毕后应输出如下格式的报告：

```
## 回归测试报告

日期：2026-03-09
执行者：Claude Code
模型：qwen3.5-plus
Cody 版本：v1.8.0

### 汇总

| 模块 | 用例 | 优先级 | 结果 | 备注 |
|------|------|--------|------|------|
| CLI | run-basic | P0 | PASS | |
| CLI | run-workdir | P0 | PASS | |
| CLI | run-verbose | P2 | FAIL | 超时，模型响应慢 |
| SDK | basic-run | P0 | PASS | |
| ... | ... | ... | ... | ... |

通过：12 / 15
失败：3 / 15
跳过：0 / 15

### 失败用例详情

> **每个 FAIL 的用例都必须记录以下信息，方便开发者定位问题。**

#### FAIL: CLI / run-verbose (TC-CLI-003)

- **预期结果**: 输出包含 Model、Workdir 等调试信息
- **实际结果**: 命令执行超时（120s 无响应），exit code 137
- **验证命令输出**:
  ```
  FAIL: no model info
  ```
- **可能原因**: 模型响应慢，网络不稳定
- **建议**: 检查 API 连通性，或切换更快的模型重试
```

> **注意**：
> - PASS 的用例只需一行汇总，不用展开
> - FAIL 的用例必须展开写「预期 vs 实际 vs 可能原因」
> - SKIP 的用例说明跳过原因（如依赖未安装、非核心用例等）
> - 如果 P0 用例 FAIL，在报告最顶部加醒目提示

## 测试用例编写规范

每个 `.md` 文件包含一个或多个测试用例，格式如下：

```markdown
## TC-{模块}-{编号}: {用例名称}

**优先级**: P0 / P1 / P2
**前置条件**: 需要什么环境 / 文件
**涉及功能**: 被测试的功能点

### 操作步骤

1. 具体的 bash 命令或 Python 代码
2. ...

### 预期结果

- 描述预期行为（宽松描述）

### 验证方法

- 具体的检查命令（文件是否存在、输出是否包含关键词等）
- AI 可直接执行的验证逻辑

### 清理

- 需要清理的临时文件 / 目录（如果有）
```

### 优先级定义

- **P0**：核心链路，必须通过（run 能执行、文件能创建）
- **P1**：重要功能，应该通过（会话管理、流式输出）
- **P2**：辅助功能，允许跳过（verbose 输出格式等）

## 注意事项

1. **API 成本**：每次完整回归会消耗 LLM token，建议在重要发版前执行
2. **超时处理**：LLM 调用可能较慢，单个用例超过 2 分钟未响应视为 FAIL
3. **环境隔离**：所有测试在 `/tmp/cody_ai_test_*` 目录下运行
4. **幂等性**：每个用例可独立执行，不依赖其他用例的执行结果
